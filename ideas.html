The Origin Story

Weldr started as "Mosaic Builder" ‚Äî an ambitious AI app builder. Some ideas worked beautifully. Others didn't survive contact with reality. Here's what we learned.

Concepts Worth Keeping

üîê Progressive Ownership

The insight: AI tools either clobber your code or you lose AI assistance entirely. Progressive ownership is the middle ground.

MANAGED AI controls completely. Can rewrite anytime.
EXTENDED Mix of AI + custom. Your @custom blocks preserved.
OWNED You control. AI can read but never overwrite.

‚ú® This concept is coming to code-chat as "Protected Files"

üìù @custom Blocks

Mark regions of code that AI must never touch. Survives regeneration. 

// AI-generated code above...

// @custom:begin user-validation
function validateUser(user) {
  // Your custom logic here
  // AI will never touch this
  return user.role === 'admin' || user.verified;
}
// @custom:end

// AI-generated code below...

The AI understands these markers and works around them. Your business logic stays yours.

üìã The Weldr Protocol

A formal, machine-enforceable system for AI-safe software evolution.

Key invariants:

  • No silent clobber: AI can never overwrite owned surfaces
  • Marker integrity: @custom blocks must stay intact
  • Loud failures: Blocked operations explain why
  • Staleness tracking: Know when your forks are outdated

This was perhaps over-engineered for an app builder, but the core ideas apply to any AI coding tool.

üéØ Surface-Level Granularity

Ownership doesn't have to be file-level. You can own:

  • FILE:components/Button.tsx ‚Äî whole file
  • BLOCK:api.ts:@custom:auth ‚Äî just one section
  • ROUTE:/api/billing/* ‚Äî all billing routes
  • ENTITY:User ‚Äî the User model everywhere

Fine-grained control means you can let AI handle boilerplate while protecting critical business logic.
